## 1. Where the repo lives & how git runs

- **Source of truth: `~/vivahOS`** (native **ext4** inside WSL). The old
  `/mnt/d/Do Not Open/vivah/vivahOS` copy is **retired** — do not run git there.
- **Git runs from WSL** on this checkout (worktree / merge / push all safe on
  ext4). The old "git-from-PowerShell only" rule was a `/mnt/d` **DrvFs**
  workaround and does **not** apply here.
- **PowerShell is used only for Windows-side things:** editing `.wslconfig` and
  `wsl --shutdown`. Never run repo git/builds from the `D:\` PowerShell prompt.
- Edit via **VS Code Remote-WSL** (`code .` from `~/vivahOS`).
- Branches: `main` (working), `master` (backup — keep in sync as the safety net).

## 2. The cache-bypass flag (important)

Use **`pnpm type-check --force`** — single dashes. This passes `--force` to
**Turbo** (bypass cache).

**Do NOT** use `pnpm type-check -- --force` — the extra `--` forwards `--force`
to `tsc`, which errors: `Compiler option '--force' may only be used with
'--build'`. (Older versions of the roadmap/prompts had this wrong; corrected.)

"Force" matters because a cached "FULL TURBO" green pass can hide a merge that
silently reverted fixes — always bypass cache when verifying a merge.

## 3. WSL memory — required for parallel sprints

**Symptom:** `db#type-check` crashes with `JavaScript heap out of memory`
(exit 134). The Drizzle schema produces a huge `tsc` type graph.

**Root cause:** WSL2 defaults to ~50% of host RAM. On a 24 GB machine WSL was
seeing only ~11 GB, and Node's default heap (~2 GB) is too small for the db package.

**Fix — two layers:**

1. Raise the WSL memory pool. Create `C:\Users\<you>\.wslconfig` (Windows side,
   no `.txt` extension):
   ```ini
   [wsl2]
   memory=16GB
   processors=8
   swap=8GB
   ```
   Then from PowerShell: `wsl --shutdown`, reopen WSL, confirm `free -h` shows
   ~15-16Gi. (Leave headroom for Windows — don't allocate all 24 GB.)

2. Raise Node's heap (persist in `~/.bashrc`):
   ```bash
   echo 'export NODE_OPTIONS=--max-old-space-size=12288' >> ~/.bashrc
   source ~/.bashrc
   ```

**Why this is non-optional for the parallel plan:** agent teams run multiple
`tsc`/build processes at once. An 11 GB pool OOMs the moment two teammates
type-check simultaneously. 16 GB pool + 12 GB heap is what makes 3-teammate
sprints safe.

## 4. `next-env.d.ts` — fresh clones fail web type-check until Next runs

**Symptom:** web type-check throws `TS2307: Cannot find module
'...couple-dusk.webp'` for every image import, even though the files exist.

**Root cause:** `apps/web/next-env.d.ts` is auto-generated by Next.js (git-ignored)
and pulls in `next/image-types/global`, the ambient declaration that types
`.webp`/image imports. A fresh clone never ran Next, so the file is absent.

**Fix — generate it once, then type-check:**
```bash
cd ~/vivahOS/apps/web
timeout 25 pnpm dev >/dev/null 2>&1 || true   # boots dev long enough to write next-env.d.ts
ls -la next-env.d.ts                            # should now exist with the /// <reference ... /> line
cd ~/vivahOS && pnpm type-check --force
```

**Sequencing fact:** on any clean checkout (including CI), boot `next dev`/`next
build` once before type-checking the web app.

## 5. Untracked / ignored assets don't travel with a clone

**Symptom:** the 5 landing `.webp` files under `apps/web/public/landing/` were
missing after clone, yet `git ls-files` listed them AND `git check-ignore`
matched them (committed via `git add -f` but still ignored).

**Effect:** `git clone` skips them (they match the ignore rule), so they only
existed in the old `/mnt/d` copy.

**Fix (already done):** copied them over. Verify bytes are real (not 0):
```bash
ls -la apps/web/public/landing/*.webp     # each several KB
```

**Latent trap to fix properly later:** "ignored **and** committed" means every
fresh clone / CI run breaks the same way. Clean fix: stop ignoring them — remove
the rule or add `!apps/web/public/landing/*.webp` to `.gitignore` — so they
travel normally. (Not urgent; note it.)

## 6. Other files that don't travel (copy from old checkout / regenerate)

Anything git-ignored is local-only. After a fresh clone, copy from the old copy
or regenerate:

- **Env files** (secrets, gitignored): `.env` (root, if present),
  `apps/api/.env`, `apps/api/.env.production`, `apps/web/.env.local`,
  `apps/ai-service/.env`. (`.env.example` templates DO come with git — ignore those.)
- **`next-env.d.ts`** — regenerate via §4.
- **`node_modules`** — run `pnpm install` (don't copy across filesystems; pnpm's
  symlink store doesn't survive a cross-fs copy).
- **`apps/ai-service/.venv`** — recreate natively (paths are baked in); do NOT
  copy from `/mnt/d`.
- **Build caches** (`.next`, `.turbo`, `dist`) — regenerated on build; clear if stale.

## 7. Local services (Docker)

```bash
docker compose up -d
docker compose ps
```

- **Postgres** (`pgvector/pgvector:pg16`, `smart-shaadi-postgres`): host port
  **5433** → container 5432. This is deliberate — a native PG18 service squats
  5432 (commit `e6f79f7`). `.env` `DATABASE_URL` must point at **5433**.
- **Redis** (`6379`), **Mongo 7** (`27017`), **Adminer** (`8080`).
- Wait for health checks: re-run `docker compose ps` until Postgres/Redis show
  **healthy** (not `health: starting`).
- If **Mongo shows unhealthy**: Postgres+Redis are what Phase-0/schema work needs;
  Mongo holds flexible profile/chat content. If a unit that touches Mongo errors:
  `docker compose restart mongo` and check `docker compose logs mongo`.
- After `wsl --shutdown`, Docker Desktop's daemon may be down — restart Docker
  Desktop (whale icon steady) and confirm WSL Integration is still enabled for
  this distro, then `docker compose up -d` again.

## 8. Prod DB migrations (unchanged by the move)

- Generate + **commit** migrations; **never** `drizzle-kit push` to prod.
- Apply prod SQL via `psql` from whichever shell reaches the Railway proxy
  (PowerShell if WSL times out on the proxy that day). Repo location is irrelevant
  to this.
- Never paste a live `DATABASE_URL` into terminal scrollback; rotate the Railway
  Postgres password if a credential ever lands somewhere shared/public.
